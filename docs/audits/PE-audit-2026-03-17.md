# PE Audit вЂ” 2026-03-17

## Context
Confidence margin downgrade implementation. Not a full audit round вЂ” append per CLAUDE.md rule.

## Change Applied

**Finding:** Confidence label inversion detected in backtest вЂ” HIGH confidence matches had lower accuracy than LOW in Premier League (HIGH=46.3% vs LOW=56.3%). Root cause: `computeConfidence` did not account for inherent uncertainty in balanced matches (low `favorite_margin`).

**Fix:** Added optional `favoriteMargin` parameter to `computeConfidence`. When a match would be rated HIGH but `favoriteMargin < MARGIN_FOR_HIGH_CONFIDENCE (0.12)`, the label degrades to MEDIUM.

## Files Modified
- `packages/prediction/src/engine/v3/constants.ts` вЂ” added `MARGIN_FOR_HIGH_CONFIDENCE = 0.12`
- `packages/prediction/src/engine/v3/confidence.ts` вЂ” added `favoriteMargin?` parameter + degradation logic
- `packages/prediction/src/engine/v3/v3-engine.ts` вЂ” compute `prelimMargin` from sorted final probs and pass to `computeConfidence`
- `packages/prediction/test/v3-engine.test.ts` вЂ” 4 new test cases for margin downgrade behavior

## Tests
- New tests: 4 (computeConfidence margin variants)
- All 6 confidence tests pass (2 pre-existing + 4 new)
- Suite: 1159 tests / 1158 passed (1 pre-existing unrelated failure: match-validator catalog size)

## Spec Alignment
- В§15: confidence computation вЂ” extended, not replaced
- Backward-compatible: `favoriteMargin` optional, existing callers unaffected
- `decision_policy_version` not bumped вЂ” `confidence` is metadata label, not policy-gated output

# SP-V4-05 вЂ” SoS Weighted Recency (Append 2026-03-17 sesiГіn 8)

## Objetivo
Ponderar los N_RECENT partidos del recency delta por la calidad del rival enfrentado.
La hipГіtesis: un partido contra el lГ­der de la liga debe pesar mГЎs que contra el colista.

## AnГЎlisis Previo (antes de implementar)

`MatchSignalRA` solo tenГ­a `{ utcDate, attack_signal, defense_signal }` вЂ” sin info del rival.
La rival_strength es computable desde `getOpponentEffective` que ya existe en В§8.
DiseГ±o elegido: embeber `rivalStrength?: number` en `MatchSignalRA` desde `computeMatchSignalsRA`.
Esto mantiene `computeRecencyDeltas` pura: no necesita contexto adicional вЂ” los pesos vienen en los signals.

## FГіrmula Implementada (В§SP-V4-05)

```
weight_i = max(0, 1 + SOS_SENSITIVITY * (rivalStrength_i в€’ 1.0))
rival_strength_i = (opp_attack_eff + opp_defense_eff) / 2

delta_attack  = ОЈ(weight_i Г— attack_signal_i)  / ОЈ(weight_i) / season_attack
delta_defense = ОЈ(weight_i Г— defense_signal_i) / ОЈ(weight_i) / season_defense
```

Fallback: `rivalStrength` undefined (opp con < RA_MIN_RIVAL_GAMES) в†’ `weight = 1.0`.
SOS_SENSITIVITY = 0 в†’ promedio uniforme bit-exacto (retrocompatibilidad completa).

## Sweep SOS_SENSITIVITY (walk-forward PD+PL+BL1, ~640 partidos evaluables)

| 0.00 | 48.28 | 49.12 | 32.18 | 0.9063    | 640    |
| 0.10 | 48.13 | 48.82 | 31.92 | 0.9019    | 640    |
| 0.15 | 48.04 | 48.52 | 31.66 | 0.8982    | 639    |

| 0.30 | 47.89 | 48.52 | 31.42 | 0.8957    | 641    |

**SOS_SENSITIVITY = 0.0 es Гіptimo.** Toda sensibilidad positiva degrada el composite score monotГіnicamente.

## ConclusiГіn вЂ” Por quГ© rival_adjustment ya captura el efecto SoS

El В§8 `computeMatchSignalsRA` normaliza cada seГ±al por los effective rates del rival:
```
attack_signal_i = goals_scored_i / rival_defense_eff_i
```
Un partido contra un rival fuerte (defense_eff alto) в†’ `attack_signal` ya se amplifica proporcionalmente.
Dar mayor peso a ese partido en el promedio = doble-conteo: el numerador ya fue ajustado por calidad del rival.

**DecisiГіn: SOS_SENSITIVITY = 0.0** (uniforme, como antes). El campo `rivalStrength` se mantiene porque:
1. Backward-compatible: SOS=0 в†’ bit-exact al estado anterior (T3-REG invariant)
2. Habilita experimentos futuros si la normalizaciГіn del rival_adjustment cambia
3. Formaliza en tests que la invariante "RA ya captura SoS" es estructuralmente correcta
