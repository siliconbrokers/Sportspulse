### Findings

#### FINDING-001 [HIGH]
- **Spec section:** Master spec §S8.2 — "Feature Flags and Shadow Mode"
- **Spec requirement:** "The feature flag controlling shadow mode activation is `NEXUS_SHADOW_ENABLED`. Format: `NEXUS_SHADOW_ENABLED=comp:football-data:PD,comp:football-data:PL`."
- **Code location:** `server/prediction/nexus-shadow-runner.ts`, line 75 — `parseNexusShadowFlag()`
- **Discrepancy:** The implementation reads `process.env.PREDICTION_NEXUS_SHADOW_ENABLED`. The spec defines the canonical env var name as `NEXUS_SHADOW_ENABLED`. The prefix `PREDICTION_` is not authorized by any spec section. Any deployment that sets `NEXUS_SHADOW_ENABLED` (the spec name) will find shadow mode permanently disabled regardless of the flag value.
- **Severity rationale:** The entire shadow mode activation mechanism is keyed to the wrong environment variable name. Any infrastructure configured per the spec will silently have shadow mode disabled. This breaks the feature flag contract defined in the master spec.
- **Minimum correction required:** Change `process.env.PREDICTION_NEXUS_SHADOW_ENABLED` to `process.env.NEXUS_SHADOW_ENABLED` in `parseNexusShadowFlag()` at line 75.

#### FINDING-002 [HIGH]
- **Spec section:** NEXUS-0 §S9.2 — "Prediction Fingerprint"
- **Spec requirement:** "Every prediction snapshot MUST include the following seven fields in its fingerprint: (1) `matchId`, (2) `buildNowUtc`, (3) `kickoffUtc`, (4) `featureSchemaVersion`, (5) `datasetWindow` (ISO interval), (6) `modelVersion`, (7) `calibrationVersion`, (8) `ensembleVersion`." (The spec lists 7 required fields plus ensembleVersion in the same normative block, for 8 total identifiable fields.)
- **Code location:** `server/prediction/nexus-shadow-runner.ts`, interface `NexusShadowSnapshot` (lines 50–70) and the return statement of `runNexusPrediction()` (lines 311–327)
- **Discrepancy:** The `NexusShadowSnapshot` interface and the persisted snapshot object are missing: `featureSchemaVersion`, `datasetWindow`, `modelVersion`, and `calibrationVersion`. Only `matchId`, `buildNowUtc`, `kickoffUtc`, and `ensembleVersion` (indirectly via `weights`) are present. Four of the eight fingerprint fields required by NEXUS-0 S9.2 are absent from every persisted snapshot.
- **Severity rationale:** The prediction fingerprint is the reproducibility and auditability contract for NEXUS predictions. Missing fields make it impossible to reconstruct the model state that produced any given prediction. This undermines the spec's explicit reproducibility guarantee (NEXUS-0 S9.3).
- **Minimum correction required:** Add `featureSchemaVersion: string`, `datasetWindow: string` (ISO interval, e.g. `"2024-08-01/2026-03-18"`), `modelVersion: string`, and `calibrationVersion: string` to the `NexusShadowSnapshot` interface and populate them in `runNexusPrediction()`. The calibration version can be pulled from `buildBootstrapCalibrationTable().calibrationVersion`. The model version should be sourced from Track 3 weights metadata or a module constant.

#### FINDING-003 [HIGH]
- **Spec section:** Market-Signal-Policy (MSP) §S2.2 — "Confidence by Capture Horizon"
- **Spec requirement:** "Track 4 confidence is determined by the distance in time from odds-capture to KICKOFF. Specifically: HIGH if captured ≤ 24h before kickoff; MEDIUM if captured between 24h and 72h before kickoff; LOW if captured > 72h before kickoff." The spec explicitly defines confidence as a function of `(kickoffUtc - snapshot_utc)`, i.e. distance from snapshot to kickoff.
- **Code location:** `packages/prediction/src/nexus/odds/canonical-serving-view.ts`, function `computeOddsConfidence()` (lines 114–129)
- **Discrepancy:** The implementation computes confidence from `snapshotAgeHours` which is the age of the snapshot relative to `buildNowUtc` (i.e. `buildNowUtc - snapshot_utc`), NOT relative to kickoff. The function's own doc comment acknowledges this: "This function uses snapshot_age_hours (age relative to buildNowUtc) as a proxy. The caller is responsible for mapping to kickoff distance if needed." However, `getCanonicalOddsSnapshot()` (the caller) does not perform this mapping — it passes the age from `buildNowUtc`, not from `kickoffUtc`. As a result, odds captured 30 hours before kickoff but consumed at `buildNowUtc` = 2 hours before kickoff are classified as HIGH (age from now = 28h → "MEDIUM" by age-threshold; age from kickoff = 30h → "HIGH" by spec). This is a systematic misclassification at the closing-line window where the spec precision matters most.
- **Severity rationale:** Confidence tier affects Track 4 activation and the data quality tier used for ensemble weight segment lookup (FULL/PARTIAL/MINIMAL). Misclassification at the closing line window — where odds carry the most information — directly corrupts the ensemble's weighting decision.
- **Minimum correction required:** `computeOddsConfidence()` must accept `kickoffUtc` as a parameter and compute age as `(kickoffUtc - snapshot_utc)` in seconds, not `(buildNowUtc - snapshot_utc)`. Propagate `kickoffUtc` from `getCanonicalOddsSnapshot()` down to this function.

#### FINDING-004 [MEDIUM]
- **Spec section:** Evaluation-and-Promotion spec §S6.2 — "Volume Conditions"
- **Spec requirement:** "Live shadow minimum: ≥ 100 predictions per production league in the live_shadow slice." This is a per-league constraint only. No aggregate-total constraint on `liveShadowN` is defined in §S6.2 or §S6.6.
- **Code location:** `packages/prediction/src/nexus/promotion/gate-evaluator.ts`, lines 131–138
- **Discrepancy:** Lines 131–138 apply a second check using `input.liveShadowN < volumeReq.minLiveShadowPerLeague` on the aggregate `liveShadowN` total (not per-league). This uses the per-league threshold (100) as a total threshold. The spec never defines an aggregate live_shadow minimum. If 3 leagues each have 99 samples (total = 297), the per-league check correctly fails. But if one league has 200 and the others have 0, the per-league check fails but the spurious aggregate check would also push `INSUFFICIENT_LIVE_SHADOW` again (idempotent due to the includes() guard). More importantly, the aggregate check can produce false negatives: if total live_shadow = 95 but per-league check already fired and was caught by the includes() guard, the finding is suppressed. The logic is incoherent and not spec-authorized.
- **Severity rationale:** Non-authorized constraint that does not match the spec's definition. Could produce duplicate or suppressed failure conditions. Does not change the gate pass/fail in most practical cases due to the `includes()` guard, but the condition is unspecified and should not exist.
- **Minimum correction required:** Remove lines 131–138 entirely. The live_shadow volume check is fully covered by the per-league loop at lines 121–129.

#### FINDING-005 [MEDIUM]
- **Spec section:** Ensemble-calibrator — `fitPerClassCalibrator()` internal structure
- **Spec requirement:** Taxonomy spec §S8.2: "Three separate one-vs-rest calibrators: one for home, one for draw, one for away. Each is an independent PAVA-fitted isotonic curve."
- **Code location:** `packages/prediction/src/nexus/ensemble/ensemble-calibrator.ts`, function `fitPerClassCalibrator()` (lines 178–202), specifically lines 198–200
- **Discrepancy:** `fitPerClassCalibrator()` computes `homePoints = fitPAVA(homePairs)` and then assigns `draw: homePoints` and `away: homePoints` — all three calibrators receive the same fitted PAVA curve computed from the same `homePairs` data. The function's own comment at lines 198–200 acknowledges: "Will be overridden per-class in fitNexusCalibration." However, `fitNexusCalibration()` (the caller) calls `fitLeagueCalibration()` which also calls `fitPerClassCalibrator()` and does not override the draw/away calibrators either — it calls `fitPAVA(homePairs)` for all three classes. The correct public interface (`fitNexusCalibrationFromTriplets`) properly separates the three classes, but `fitNexusCalibration()` does not use it and is therefore broken for one-vs-rest calibration. Since `runNexusPrediction()` uses `buildBootstrapCalibrationTable()` (bootstrap identity), this defect is not exercised in production Phase 4, but it renders the `fitNexusCalibration()` path non-compliant.
- **Severity rationale:** The `fitNexusCalibration()` function is spec-defined as a calibration entry point. Its current implementation silently produces three identical calibrators instead of three independent one-vs-rest calibrators. Any caller using `fitNexusCalibration()` (non-triplets variant) would receive wrong calibration without any error or warning.
- **Minimum correction required:** Either: (a) delete `fitNexusCalibration()` entirely and mandate callers use `fitNexusCalibrationFromTriplets()`, adding an explicit deprecation comment; or (b) fix `fitNexusCalibration()` to accept class-separated data and call `fitPAVA` independently per class. The broken function should at minimum throw `NotImplementedError` to prevent silent misuse.

#### FINDING-006 [MEDIUM]
- **Spec section:** Master spec §S1.3 — "V3 Coexistence" and §S8.5 — "No Shared Mutable State"
- **Spec requirement:** "NEXUS maintains its own Elo pool, separate from V3's pool." (Taxonomy spec S2.1, master spec S8.5). The canonical match data (schedules, scores) is the only shared resource; all derived state must be isolated.
- **Code location:** `server/prediction/nexus-shadow-runner.ts`, lines 196–203 and line 214
- **Discrepancy:** `isNeutralVenue` is hardcoded to `false` for all matches: "canonical data does not provide neutral_venue flag." The comment is correct that the `DataSource.getMatches()` response does not expose `neutral_venue`. However, taxonomy spec S3.2 states: "Neutral venue flag MUST come from canonical match data — never inferred or defaulted." The code explicitly defaults a spec-required flag to `false` (home team always has advantage), which constitutes an inference (or rather, a hard-coded assumption). This is particularly consequential for international competition matches that may be played at neutral venues.
- **Severity rationale:** The spec explicitly prohibits defaulting `isNeutralVenue` — it must come from canonical data. The workaround comment acknowledges the data gap but the spec does not authorize a fallback. The track1 output for all matches will over-apply home advantage, biasing all predictions.
- **Minimum correction required:** Add `neutral_venue: boolean` to the `DataSource.getMatches()` return type (or the match shape used in the shadow runner) and populate it from the canonical match store. Until canonical data exposes this flag, the runner should either (a) skip matches where `neutral_venue` cannot be determined with certainty, or (b) formally request a spec amendment authorizing a conservative default. The current silent default violates the spec's prohibition.

#### FINDING-007 [LOW]
- **Spec section:** NEXUS-0 §S9.2 — Prediction Fingerprint / Master spec §S8.2
- **Spec requirement:** Predictions must carry a `learnedAt` timestamp that reflects the actual time the weights were trained.
- **Code location:** `server/prediction/nexus-shadow-runner.ts`, function `buildBootstrapWeightRegistry()`, line 417
- **Discrepancy:** `learnedAt: '2026-03-18T00:00:00.000Z'` is hardcoded as a static string. When the server starts on 2026-03-20 or any future date, the bootstrap registry will still carry the 2026-03-18 timestamp. For bootstrap weights, this is a naming/accuracy concern rather than a correctness concern (the bootstrap weights are known constants, not trained), but the field semantics imply "when was this model trained" and the hardcoded past date will become increasingly misleading.
- **Severity rationale:** LOW because the bootstrap weights themselves are correct; only the `learnedAt` timestamp is stale. Does not affect prediction accuracy. Affects auditability.
- **Minimum correction required:** Replace the hardcoded timestamp with `new Date().toISOString()` at the time `buildBootstrapWeightRegistry()` is called, or use a module-level constant `NEXUS_BOOTSTRAP_VERSION_DATE` that can be updated when bootstrap weights change.

#### FINDING-008 [LOW]
- **Spec section:** Master spec §S8.5 — V3 Isolation; NEXUS-0 §S3.2 — Anti-Lookahead
- **Spec requirement:** "All NEXUS modules must be independent from V3. No NEXUS module imports from `engine/v3/` or any V3 module." (Master spec S8.4, S8.5). Track 3 spec S2.2: "independence of tracks — no import from engine/v3/."

This bypasses the `@sportpulse/prediction` workspace package boundary. ### Audit Notes

**Spec ambiguity — Prediction fingerprint field count (NEXUS-0 §S9.2):**
Section S9.2 lists "7 required fields" in its title but the normative list contains 8 identifiable fields including `ensembleVersion`. The implementation includes `ensembleVersion` (via the weight registry) but not `calibrationVersion`, `modelVersion`, `featureSchemaVersion`, or `datasetWindow`. Even under a strict 7-field reading, at least 3 of these are clearly missing. A spec clarification on the exact enumeration would be useful.

**Spec ambiguity — `fitNexusCalibration()` vs `fitNexusCalibrationFromTriplets()` (Taxonomy §S8.2):**
The spec defines "one calibrator per class (home, draw, away)" but does not specify the calling convention for passing class-separated data. The code provides two entry points with inconsistent behavior. The spec should specify whether the input is unified or pre-separated by class. The `fitNexusCalibrationFromTriplets()` function is the correct implementation; the `fitNexusCalibration()` function should either be fixed or declared deprecated.

**Spec gap — isNeutralVenue in canonical match data (Taxonomy §S3.2):**
The spec prohibits defaulting `isNeutralVenue` to any value, but the canonical match store (`DataSource.getMatches()`) does not expose this field. Either the canonical data spec (sports data normalization spec) needs to include `neutral_venue` as a required field, or the prediction spec needs to define an explicit fallback policy for domestic league matches (where neutral venue is extremely rare). Without spec guidance, any implementation choice violates one spec or the other.

**Phase 4 scope note:**
Several findings (FINDING-001, FINDING-002, FINDING-003) represent spec violations that are currently non-observable in production because Track 4 is hardcoded DEACTIVATED and no shadow snapshots are being read by the public API. However, they remain spec violations that will become observable as NEXUS progresses toward promotion. The audit reports them at their correct severity as pre-production issues that must be resolved before the live shadow phase (evaluation spec §S6.6) can be claimed as complete.

**Finding closed by:** domain-contracts-agent (Claude Sonnet 4.6)

**Spec requirement (MSP S2.2):** Track 4 confidence is determined by `(kickoffUtc - snapshot_utc)` — distance from capture to kickoff, NOT from `buildNowUtc`.

**Changes applied:**
