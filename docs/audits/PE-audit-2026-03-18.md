### Spec Ambiguities Resolved

1. **Logistic missingness policy (taxonomy S5.4b):** Spec says "feature excluded from input vector entirely" for logistic regression with missing values. Implementation uses 0 as neutral value for the corresponding weight term (mathematically equivalent to "not contributing"). Documented explicitly — this is NOT global mean imputation.

2. **Confidence thresholds:** Spec (S5.6) describes degradation conditions but gives no specific thresholds for HIGH/MEDIUM/LOW confidence. Resolution: HIGH ≥ 7/9 features present, MEDIUM 4–6, LOW < 4. The key insight is that with even 1 historical match, nearly all 9 features are computable (formGeneral uses team's own prior per S5.6), so LOW confidence effectively means zero history.

3. **H2H matching direction:** Spec refers to H2H without specifying if direction matters. Resolution: encounters are matched bidirectionally (A vs B OR B vs A). Win attribution is always from the "home" team's perspective for the current prediction.

### Coverage Notes

- Travel distance feature (mentioned in task prompt) is NOT in taxonomy spec S5.3.1 eligible features list. It was not implemented to avoid spec-unauthorized additions.
- `match_importance` (scalar from prompt) is implemented as `computeMatchImportanceScore` utility but is NOT a direct spec feature — it is derived from `competitiveImportance` + `seasonPhase` which ARE in the spec. The utility is available for future model extensions.

### Invariants Verified

1. **Pre-kickoff guard (S6.2):** `appendScorecardEntry` silently ignores entries where `predictionUtc >= kickoffUtc`. Shadow runner also enforces this at the eligibility filter (`m.startTimeUtc > buildNowUtc`) and as a defense-in-depth check before writing.
2. **Idempotency (S11.3 immutability):** `appendScorecardEntry` checks file existence before writing. If file exists, existing entry is preserved — never overwritten.
3. **Disjoint invariant (S12.8):** `buildCombinedScorecard` throws `Error` with message "Disjoint invariant violated" and lists overlapping matchIds if any matchId appears in both HWF and LS.
4. **Cardinality invariant (S12.8):** `combined.n = hwf.n + ls.n` — enforced by construction (entries are the disjoint union).
5. **Weighted average RPS:** `combined.rps_mean = (n_hwf * rps_hwf + n_ls * rps_ls) / (n_hwf + n_ls)` — per spec S5.2.7.
6. **V3 isolation (S8.4, S8.5):** Shadow runner never imports from V3 pipeline. No shared mutable state. Shadow output goes to `cache/nexus-shadow/` (separate from V3's storage).
7. **Fault isolation:** Shadow runner wraps every match in try/catch. Outer function never throws. Server never crashes from NEXUS failures.
8. **Atomic writes:** Both `scorecard-store.ts` and `nexus-shadow-runner.ts` use `.tmp → rename` pattern (same as matchday-cache). Cleanup on write failure.
9. **API isolation:** Shadow runner exposes no routes. Zero impact on `/api/ui/*` endpoints.

### Shadow Runner Design Notes

- Phase 4 uses bootstrap Track 1+2 (Poisson from canonical DataSource) and deactivated Track 3/4.
- The ensemble runs in LIMITED_MODE (Track 3 null → taxonomy spec S9.3).
- Weight registry uses bootstrap values (track12=1.0, track3=0.0, track4=0.0) until walk-forward training accumulates.
- Calibration uses bootstrap table (identity-like) until ≥300 samples.
- As real live_shadow predictions accumulate, the training pipeline will replace bootstrap values with learned weights.
- Flag: `PREDICTION_NEXUS_SHADOW_ENABLED=comp:football-data:PD,comp:football-data:PL,...` (optional).
