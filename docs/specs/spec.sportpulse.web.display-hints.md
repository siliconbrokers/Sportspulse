# SportPulse — Display Hints Specification (Chips + Labels)

Version: 1.1  
Status: Draft (implementation-ready)  
Scope: Backend-generated, versioned display hints for “dummies” UI (chips/labels/explain lines) derived deterministically from snapshot truth, including **match-first cards** for football UX  
Audience: Backend, Frontend, QA, AI-assisted development workflows

---

## 1. Purpose

This spec defines a **robust** way to render human-friendly UI labels (e.g., `🔥 Picante`, `⏳ Hoy · en 7 h`, `📅 En 4 días`) **without** moving semantic logic into the frontend.

Principles:

- Backend produces **deterministic display hints** as part of the snapshot artifact.
- Frontend renders them as-is (no recalculation, no threshold drift).
- Display behavior is **versioned** so historical snapshots remain interpretable.
- For football UX, “cards” must be **match-first**: **1 partido = 1 tarjeta**, with both teams’ form chips inside the same card.

Display hints are a *presentation mapping layer* that sits **after** scoring/layout and does **not** affect scoring.

---

## 2. Non-goals

- Display hints do **not** change scoring, ordering, or geometry of treemap tiles.
- Display hints are **not** generated by an LLM.
- Display hints must not introduce new scoring signals; they only map existing truth to human labels.
- This spec does **not** redefine policy weights or signal semantics.

---

## 3. Where this lives in the pipeline

In `packages/snapshot` (or a dedicated `packages/presentation` later), insert a stage:

**canonical → signals → scoring → layout → snapshot DTO assembly → display hints mapping → serialize**

Display hints mapping consumes only:
- existing score/signal outputs
- lifecycle status (optional)
- `buildNowUtc`
- (optional) previous snapshot for delta chips

It must not call provider or recompute signals.

---

## 4. Versioning

Add a display rules identity to the snapshot metadata:

- `displayRulesKey` (string)
- `displayRulesVersion` (number)

Any material change to:
- thresholds
- wording (“Picante”)
- icon choices
- formatting rules (hours vs days)
requires bumping `displayRulesVersion`.

Independent from:
- `policyVersion` (scoring)
- `layoutAlgorithmVersion` (geometry)
- `snapshotSchemaVersion` (DTO shape)

Recommended:
- `displayRulesKey = "sportpulse.display.dummies"`

## 5. DTOs

### 5.1 DisplayRulesDTO (snapshot-level metadata)

```ts
type DisplayRulesDTO = {
  displayRulesKey: string;      // e.g. "sportpulse.display.dummies"
  displayRulesVersion: number;  // integer
};
```

### 5.2 DisplayChipDTO (generic chip)

```ts
type ChipLevel = "INFO" | "OK" | "WARN" | "HOT" | "ERROR" | "UNKNOWN";

type DisplayChipDTO = {
  icon: string;        // emoji or icon token; keep stable per version
  label: string;       // short user-facing label
  level: ChipLevel;    // styling/intensity bucket (optional)
  kind: string;        // stable discriminator, e.g. "FORM_HOT", "TIME_TODAY_HOURS"
};
```

### 5.3 ExplainLineDTO (optional one-line explanation)

```ts
type ExplainLineDTO = {
  text: string;        // template-driven, stable
  kind: string;        // e.g. "WHY_MATCH_SIMPLE"
};
```

## 8. MatchCardDTO (match-first UI contract)

Match cards are a **projection** derived deterministically from canonical match schedule + team truth.

```ts
type MatchCardDTO = {
  matchId: string;
  kickoffUtc?: string;                  // optional if missing; then timeChip must be TIME_UNKNOWN
  status?: "SCHEDULED" | "LIVE" | "FINISHED" | "UNKNOWN";

  timeChip: DisplayChipDTO;             // ⏳ Hoy · en 7 h, 📅 En 4 días, 🔴 En juego, ⚠️ Sin fecha

  home: {
    teamId: string;
    name: string;
    crestUrl?: string;
    formChip?: DisplayChipDTO;          // 🔥 Picante / ✅ Viene bien / ➖ Normal / ❌ Viene mal / ⚠️ Sin datos
  };

  away: {
    teamId: string;
    name: string;
    crestUrl?: string;
    formChip?: DisplayChipDTO;
  };

  // Optional: for sorting and debugging (NOT shown as raw numbers in UI)
  rankScore?: number;                   // derived deterministically; see section 8.2
  explainLine?: ExplainLineDTO;         // optional
  warnings?: WarningDTO[];              // optional match-specific warnings
};
```

### 8.1 MatchCard dedupe rule
- Exactly **one** MatchCard per `matchId`.
- If you encounter multiple team-derived records referring to the same match, you must **merge** into the same card.

### 8.2 MatchCard ranking (deterministic)
To avoid the old “two cards for one match” problem, the match card must have a single ranking value.

Default MVP rule (simple and stable):
- `rankScore = 1 - (1 - homeTeam.displayScore) * (1 - awayTeam.displayScore)`
  - union-probability style: increases when either team is high, saturates gracefully.

Fallbacks:
- if one team missing → use the available team displayScore
- if both missing → rankScore = 0

**Important:** `rankScore` is used only for ordering match cards. The UI must not show the raw number.

## 9. Mapping rules (displayRulesKey=sportpulse.display.dummies, v1)

### 9.1 Form chip mapping (TEAM → MatchCard/home-away)

Inputs:
- `rawPoints` in last 5 (0..15) if available (preferred)
- else `normForm` (0..1)

Rules:
- missing → `⚠️ Sin datos` (level=UNKNOWN, kind=FORM_MISSING)

If rawPoints exists:
- 12–15 → `🔥 Picante` (level=HOT, kind=FORM_HOT)
- 8–11  → `✅ Viene bien` (level=OK, kind=FORM_GOOD)
- 5–7   → `➖ Normal` (level=INFO, kind=FORM_NORMAL)
- 0–4   → `❌ Viene mal` (level=WARN, kind=FORM_BAD)

If only normForm:
- >= 0.80 → HOT
- >= 0.53 → GOOD
- >= 0.33 → NORMAL
- < 0.33  → BAD

### 9.2 Time chip mapping (MATCH)

Input:
- `hoursUntilKickoff`
- `status` (LIVE optional)

Rules:
- missing kickoff/unknown → `⚠️ Sin fecha` (level=UNKNOWN, kind=TIME_UNKNOWN)
- LIVE → `🔴 En juego` (level=HOT, kind=TIME_LIVE)
- hours <= 0 and not LIVE → `⏱️ Ya empezó` (level=WARN, kind=TIME_STARTED)

Formatting:
- 0 < hours < 24:
  - icon=`⏳`, label=`Hoy · en {H} h`, kind=TIME_TODAY_HOURS
  - H = ceil(hours)
- 24 <= hours < 48:
  - icon=`⏳`, label=`Mañana · en {H} h`, kind=TIME_TOMORROW_HOURS
  - H = ceil(hours)
- 48 <= hours <= 168:
  - icon=`📅`, label=`En {D} días`, kind=TIME_DAYS
  - D = round(hours/24)
- hours > 168:
  - icon=`🗓️`, label=`En {D} días`, kind=TIME_LATER_DAYS
  - D = round(hours/24)

### 9.3 Delta chip mapping (optional movers)

Input:
- `deltaDisplayScore = current.displayScore - prev.displayScore`

Rules:
- if prev missing → omit deltaChip
- delta >= 0.15 → `🔺 Subió mucho` (level=HOT, kind=DELTA_UP_BIG)
- delta >= 0.07 → `🔺 Subió` (level=OK, kind=DELTA_UP)
- -0.07 < delta < 0.07 → omit
- delta <= -0.07 → `🔻 Bajó` (level=WARN, kind=DELTA_DOWN)
- delta <= -0.15 → `🔻 Bajó mucho` (level=ERROR, kind=DELTA_DOWN_BIG)

### 9.4 Explain lines (optional, template-driven)

Never LLM.

Match example:
- `Porque: {timeChip.label} + {homeFormShort} / {awayFormShort}`

Where `homeFormShort` is one of:
- `local picante` / `local viene bien` / `local normal` / `local viene mal` / `local sin datos`

and similarly for away.

## 10. Testing requirements (must-have)

### 10.1 Golden fixtures
For each golden fixture F1–F6, expected snapshots must include:
- `displayRules` metadata
- deterministic chips for relevant teams
- deterministic `matchCards` with:
  - `timeChip`
  - both teams’ `formChip`
  - dedupe guarantee (1 card per matchId)

### 10.2 Acceptance matrix mapping
Extend contract tests to ensure:
- matchCards are deduped by matchId
- frontend does not recompute chips

## 12. Implementation note (rawPoints source)

Best practice:
- Store `rawPoints` and `matchesUsed` in `SignalDTO.params` for `FORM_POINTS_LAST_5`.
- Display mapping reads those params for robust form-chip mapping.

Avoid:
- reconstructing raw points from norm without `maxPoints`.

---

## 13. Summary

Versioned backend display hints produce a “for dummies” UX while preserving determinism and testability. Match-first cards eliminate duplicate team-based cards by merging home/away into a single card per match, while still showing both teams’ form state and a single match time chip. Frontend renders; backend decides mapping.
